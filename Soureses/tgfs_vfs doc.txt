1. REWOLUCJA BRAKU STRUKTURY DRZEWIASTEJ (NO-DIRECTORY TRAVERSAL)
------------------------------------------------------------------
Tradycyjne systemy plików (NTFS, ext4, FAT32) trzymają zasoby w hierarchicznych 
strukturach folderów. Aby otworzyć ikonę lub uruchomić program, system musi 
tekstowo parsować i odczytywać z dysku każdy kolejny folder w ścieżce 
(np. /usr/share/gui/icon.png), generując powolne operacje wejścia/wyjścia (I/O).

System TGFS likwiduje folderowy paradygmat na rzecz całkowicie płaskiej 
przestrzeni zorientowanej na tagi bitowe. Wszystkie zasoby i pliki komputera 
zestawione są w jednej globalnej tablicy indeksów — TAG REGISTRY. 
Czas odszukania dowolnego pliku jest stały, niezależny od wielkości dysku 
i wynosi zero operacji tekstowych.


2. ANATOMIA STRUKTURY REJESTRU TAGÓW (64-BYTE RECORD SCHEME)
------------------------------------------------------------
Tag Registry leży pod adresem zapisanym w Superblocku (Sektor 1 dysku). 
Pojedynczy sektor dysku (512 bajtów) mieści dokładnie 8 deskryptorów plików. 
Pojedynczy deskryptor zajmuje sztywną przestrzeń 64 bajtów:

  - Offset 0  (4 bajty):  Unikalne ID pliku (File ID, 32-bit dword).
                          Wartość 0x00000000 flaguje rekord jako pusty/wolny.
  - Offset 4  (8 bajtów): 64-bitowa maska cech i binarnego formatu (Tag Mask).
  - Offset 12 (8 bajtów): 64-bitowy fizyczny sektor startowy danych na dysku (LBA).
  - Offset 20 (8 bajtów): Dokładny rozmiar pliku w bajtach (64-bit integer).
  - Offset 28 (36 bajtów): Nazwa pliku w standardzie ASCII (Wyłącznie dla celów GUI).


3. SPECYFIKACJA POLIMORFICZNYCH TAGÓW FORMATÓW (POLYNOMIAL BITMASK)
-------------------------------------------------------------------
Filtrowanie typów plików w funkcji `tgfs_find_files_by_tag` realizowane jest 
poprzez operację logiczną AND bezpośrednio na procesorze. Struktura maski:

  - Bit 0  (0x0000000000000001) -> TAG_SYSTEM (Zasoby jądra, tablice IDT/GDT, fonty)
  - Bit 1  (0x0000000000000002) -> TAG_GUI (Zasoby pętli graficznej okien)
  - Bit 2  (0x0000000000000004) -> TAG_APPLICATION (Pliki binarne kodu programów)
  - Bit 3  (0x0000000000000008) -> TAG_IMAGE (Surowe dane grafiki monitora TrueColor)
  - Bit 16 (0x0000000000010000) -> TAG_FOREIGN_ELF (Obcy format Linux ELF64)
  - Bit 17 (0x0000000000020000) -> TAG_FOREIGN_EXE (Obcy format Windows PE/EXE)


4. OPERACYJNE DZIAŁANIE WARSTWY ZGODNOŚCI (ZERO-CRASH EMULATION)
-----------------------------------------------------------------
Aby Twój system bez problemów otwierał pliki ze starych i ciężkich systemów 
operacyjnych (Windows/Linux), JIT-Mapped Poly-Loader wykonuje następujący łańcuch:

  A. Identyfikacja Taga Formatowego:
     Podczas wywołania `tgfs_load_and_map_file`, loader sprawdza bity 16 i 17. 
     Zamiast uruchamiać potężne parsery, automatycznie rozpoznaje układ pliku, 
     wyciąga z nagłówka logiczny punkt startowy i ustawia go jako Entry Point.

  B. Przechwytywanie Mostu Sprzętowego (Syscall Intercept):
     Obca aplikacja uruchomiona w Twoim Schedulerze w pewnym momencie wyśle rozkaz 
     `syscall`, oczekując interfejsu Linuxa lub Windowsa. Rejestr MSR procesora 
     przekierowuje to zapytanie do `syscall_compatibility_layer`. 
     Jądro przechwytuje zapytanie o RAM czy grafikę i w ułamku mikrosekundy 
     tłumaczy je na Twój sprzęt (GOP HDMI / krzemowa bitmapa PMM), oddając 
     odpowiedź do programu. Obcy program działa bezbłędnie i stabilnie, 
     wykorzystując pełną wydajność Twojego jądra.
